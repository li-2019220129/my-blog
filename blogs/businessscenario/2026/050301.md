前端这些年一直在讲组件化，但很多时候我们说的“组件”，其实是 React 组件、Vue 组件，或者某个框架内部的组件模型。Web Components 不一样，它不是某个框架的私有能力，而是一组浏览器原生提供的标准，目标很明确：让开发者可以像使用原生标签一样，定义自己的组件。

如果把它说得更直接一点，Web Components 解决的是两个长期存在的问题：

- 组件能不能脱离框架存在
- 样式和结构能不能真正做到隔离

这也是它一直被设计系统、跨团队组件库、低耦合嵌入式 UI 场景看重的原因。

## 一、Web Components 到底是什么

Web Components 不是某一个单独 API，而是几项能力组合起来形成的标准体系，核心通常包括下面三个部分：

### 1. Custom Elements

`Custom Elements` 用来定义自定义标签，比如：

```html
<user-card></user-card>
<price-tag></price-tag>
<app-dialog></app-dialog>
```

浏览器在识别到这些标签后，会把它们和你定义的类绑定起来，从而拥有自己的行为、生命周期和内部逻辑。

### 2. Shadow DOM

`Shadow DOM` 用来做封装。组件内部的 DOM 结构和样式可以被隔离起来，不会轻易受到外部页面 CSS 的污染，组件内部的样式也不会反过来影响外部页面。

这点和我们平时说的“样式 scoped”很像，但 Shadow DOM 是浏览器级别的隔离，不是编译器模拟出来的局部作用域。

### 3. HTML Template / Slot

`template` 用来声明不会立即渲染的 DOM 片段，适合做组件模板。

`slot` 则提供内容分发能力，让组件可以预留插槽位置，允许使用者把内容传进去。这和 Vue 的插槽、React 的 `children` 在思想上很接近，只不过这里是浏览器原生机制。

很多资料会把 `ES Modules` 也一起提到，但严格来说，它更像是现代 Web Components 的常用配套能力，而不是标准定义的一部分。

## 二、为什么 Web Components 这几年又被重新重视

早些年大家提到 Web Components，常见印象是“标准很好，但不好用”。这并不完全错，因为在那个阶段，生态、浏览器实现、工具链支持都不够成熟。

但现在环境已经变了，原因主要有几点：

### 1. 跨框架复用的价值变高了

很多公司内部不是只有一个技术栈。某些老项目是 Vue 2，新项目用 React，后台系统可能还是 Angular，甚至还有一部分页面是原生或服务端模板直出。

这时候如果你想做一套真正通用的基础组件，比如：

- 日期选择器
- 文件上传器
- 富文本输入框
- 组织架构选择器

那么基于某个框架独占语法写出来的组件，复用边界就会很明显。Web Components 的优势恰恰在这里：只要对方能写 HTML，就能接入。

### 2. 浏览器原生封装能力足够强

以前大家做样式隔离，常见手段是：

- BEM 命名
- CSS Modules
- scoped CSS
- iframe 隔离

这些方案都能解决一部分问题，但都不是浏览器原生语义。Shadow DOM 提供的是更底层的封装边界，对于组件库作者来说，稳定性会更高。

### 3. 适合做设计系统和基础设施层组件

如果一个组件要在多个产品线、多个前端栈、多个微前端子应用之间共享，那么“框架中立”往往比“开发体验最顺手”更重要。Web Components 在这个层面非常合适。

## 三、先看一个最小可用示例

下面先写一个最基础的 Web Component，感受一下它的使用方式。

```js
class HelloCard extends HTMLElement {
  constructor() {
    super();
  }

  connectedCallback() {
    this.innerHTML = `
      <div>
        <h3>Hello Web Components</h3>
        <p>这是一个最简单的自定义元素。</p>
      </div>
    `;
  }
}

customElements.define("hello-card", HelloCard);
```

页面里直接使用：

```html
<hello-card></hello-card>
```

这已经是一个合法的 Web Component 了，但它还有几个明显问题：

- 模板直接写在 `connectedCallback` 里，可维护性差
- 没有样式隔离
- 没有属性与状态设计
- 没有事件对外通信

所以真实项目里，我们通常会再往前走一步，引入 Shadow DOM 和更完整的组件结构。

## 四、Custom Elements：自定义元素的核心机制

### 1. 命名规则

自定义元素名称必须包含连字符，比如：

```html
<user-card></user-card>
```

不能写成：

```html
<usercard></usercard>
```

这个规则是为了和原生 HTML 标签区分开，避免未来标准扩展时发生命名冲突。

### 2. 定义方式

```js
class UserCard extends HTMLElement {}

customElements.define("user-card", UserCard);
```

这里的 `customElements.define()` 可以理解成注册动作。注册完成后，浏览器遇到 `<user-card>` 时，才知道应该实例化哪个类。

### 3. 生命周期

自定义元素最常用的生命周期有这几个：

- `constructor`
- `connectedCallback`
- `disconnectedCallback`
- `attributeChangedCallback`
- `adoptedCallback`

#### constructor

构造函数里适合做基础初始化，比如创建 Shadow Root、绑定实例字段。这里不要过早依赖外部 DOM，因为组件此时可能还没插入文档。

```js
constructor() {
  super();
  this.attachShadow({ mode: "open" });
}
```

#### connectedCallback

组件插入到文档时触发。常见用途包括：

- 首次渲染
- 绑定事件
- 发起初始数据请求

#### disconnectedCallback

组件从文档中移除时触发。适合做清理工作，比如：

- 移除事件监听
- 清理定时器
- 释放订阅

#### attributeChangedCallback

当监听的属性变化时触发。它是 Web Components 里连接“外部传参”和“内部更新”的关键入口。

```js
static get observedAttributes() {
  return ["name"];
}

attributeChangedCallback(name, oldValue, newValue) {
  if (name === "name" && oldValue !== newValue) {
    this.render();
  }
}
```

#### adoptedCallback

当节点被移动到新的 `document` 时触发，这个场景相对少见，实际业务里用得不多。

## 五、Shadow DOM：真正的封装边界

如果说 Custom Elements 解决的是“如何定义一个组件”，那么 Shadow DOM 解决的就是“这个组件如何独立存在”。

### 1. 创建 Shadow Root

```js
class UserCard extends HTMLElement {
  constructor() {
    super();
    this.attachShadow({ mode: "open" });
  }
}
```

`mode` 有两个值：

- `open`：可以通过 `element.shadowRoot` 从外部访问
- `closed`：外部无法直接拿到 `shadowRoot`

大多数业务组件会优先使用 `open`，因为调试、测试、排查问题更方便。

### 2. 样式隔离

```js
this.shadowRoot.innerHTML = `
  <style>
    .title {
      color: #1f2937;
      font-size: 18px;
    }
  </style>
  <h3 class="title">用户信息</h3>
`;
```

这个 `.title` 的样式只在当前 Shadow DOM 内部生效，不会影响页面上别的 `.title`，页面外部定义的 `.title` 也通常不会污染进来。

这种隔离对组件库非常重要，因为你无法假设接入方的页面样式是干净的。只要页面规模一大，类名冲突几乎是必然事件。

### 3. 事件传播需要理解 retargeting

在 Shadow DOM 里，事件并不是简单地“照常冒泡”这么直接。浏览器会做事件重定向，也就是 `event.target` 在穿透 Shadow 边界时可能会被改写，从而保护内部结构。

这意味着如果你在组件外部监听事件，看到的目标节点不一定是内部真实点击的那个元素。

如果要主动向外派发事件，建议显式使用：

```js
this.dispatchEvent(
  new CustomEvent("change", {
    detail: { value: "new value" },
    bubbles: true,
    composed: true
  })
);
```

其中：

- `bubbles: true` 表示允许冒泡
- `composed: true` 表示允许穿过 Shadow DOM 边界

如果漏掉 `composed: true`，外层应用有时就接不到这个事件。

## 六、Template 和 Slot：让组件结构更灵活

### 1. template 的作用

`template` 标签中的内容默认不会直接渲染，适合用来保存组件模板。

```html
<template id="user-card-tpl">
  <style>
    .card {
      padding: 16px;
      border: 1px solid #dcdfe6;
      border-radius: 8px;
    }
  </style>
  <div class="card">
    <h3 class="name"></h3>
    <p class="role"></p>
  </div>
</template>
```

然后在组件里克隆它：

```js
const tpl = document.getElementById("user-card-tpl");
const content = tpl.content.cloneNode(true);
this.shadowRoot.appendChild(content);
```

这种写法比把整段 HTML 字符串硬编码在 JS 里更清晰，尤其是在模板较复杂的时候。

### 2. slot 的作用

有些组件不应该把内容全部写死，而是要允许外部传入局部内容，这时就需要插槽。

```js
this.shadowRoot.innerHTML = `
  <style>
    .panel {
      border: 1px solid #e5e7eb;
      border-radius: 10px;
      padding: 16px;
    }
    .header {
      margin-bottom: 12px;
      font-weight: 600;
    }
  </style>
  <div class="panel">
    <div class="header">
      <slot name="title">默认标题</slot>
    </div>
    <div class="body">
      <slot></slot>
    </div>
  </div>
`;
```

外部使用：

```html
<app-panel>
  <span slot="title">订单详情</span>
  <p>这里是面板内容。</p>
</app-panel>
```

这类能力在弹窗、卡片、表单容器、提示框等场景里非常常见。

## 七、一个更接近生产可用的例子

下面写一个稍微完整一点的 `user-card`，包含：

- 属性驱动渲染
- Shadow DOM 封装
- 对外派发事件
- 基础样式隔离

```js
class UserCard extends HTMLElement {
  static get observedAttributes() {
    return ["name", "role", "status"];
  }

  constructor() {
    super();
    this.attachShadow({ mode: "open" });
    this.handleClick = this.handleClick.bind(this);
  }

  connectedCallback() {
    this.render();
    this.shadowRoot
      .querySelector(".action")
      ?.addEventListener("click", this.handleClick);
  }

  disconnectedCallback() {
    this.shadowRoot
      .querySelector(".action")
      ?.removeEventListener("click", this.handleClick);
  }

  attributeChangedCallback(name, oldValue, newValue) {
    if (oldValue !== newValue && this.isConnected) {
      this.render();
    }
  }

  get name() {
    return this.getAttribute("name") || "未命名用户";
  }

  get role() {
    return this.getAttribute("role") || "未知角色";
  }

  get status() {
    return this.getAttribute("status") || "offline";
  }

  handleClick() {
    this.dispatchEvent(
      new CustomEvent("view-profile", {
        detail: {
          name: this.name,
          role: this.role
        },
        bubbles: true,
        composed: true
      })
    );
  }

  render() {
    const badgeClass =
      this.status === "online" ? "badge badge--online" : "badge";

    this.shadowRoot.innerHTML = `
      <style>
        :host {
          display: block;
          font-family: Arial, sans-serif;
        }

        .card {
          border: 1px solid #dcdfe6;
          border-radius: 12px;
          padding: 16px;
          background: #fff;
          box-shadow: 0 8px 24px rgba(15, 23, 42, 0.06);
        }

        .name {
          margin: 0 0 8px;
          font-size: 18px;
          color: #111827;
        }

        .role {
          margin: 0 0 12px;
          color: #4b5563;
        }

        .badge {
          display: inline-block;
          padding: 4px 10px;
          border-radius: 999px;
          background: #f3f4f6;
          color: #374151;
          font-size: 12px;
        }

        .badge--online {
          background: #dcfce7;
          color: #166534;
        }

        .action {
          margin-top: 14px;
          border: 0;
          border-radius: 8px;
          padding: 8px 14px;
          background: #111827;
          color: #fff;
          cursor: pointer;
        }
      </style>

      <div class="card">
        <h3 class="name">${this.name}</h3>
        <p class="role">${this.role}</p>
        <span class="${badgeClass}">${this.status}</span>
        <div>
          <button class="action">查看资料</button>
        </div>
      </div>
    `;
  }
}

customElements.define("user-card", UserCard);
```

使用方式：

```html
<user-card
  name="李志扬"
  role="Frontend Engineer"
  status="online"
></user-card>

<script>
  document.querySelector("user-card").addEventListener("view-profile", (event) => {
    console.log(event.detail);
  });
</script>
```

这个例子已经比较接近真实业务组件的基本形态了。

## 八、属性、属性值、状态，该怎么设计

Web Components 在工程实践里最容易写乱的地方，不是语法，而是组件接口设计。尤其是下面这几个概念，很多人一开始会混在一起：

- HTML Attributes
- DOM Properties
- Internal State

### 1. Attributes

也就是写在标签上的内容，例如：

```html
<user-card name="张三" status="online"></user-card>
```

它的优点是直观，适合声明式使用，也方便在非框架环境中直接书写。

### 2. Properties

也就是 DOM 实例上的属性：

```js
const card = document.querySelector("user-card");
card.user = {
  name: "张三",
  role: "Frontend Engineer"
};
```

如果你传的是对象、数组、函数，通常更适合走 property，而不是 attribute。因为 attribute 本质上是字符串通道。

### 3. Internal State

组件内部状态不应该全部暴露给外部。比如一个下拉组件内部的：

- 是否展开
- 当前 hover 项
- 输入框临时关键字

这些都更适合留在组件内部维护。

### 4. 一个实用建议

如果你的组件要长期维护，建议一开始就约定好：

- 简单、可序列化的数据走 attribute
- 复杂对象走 property
- 内部交互过程走 state
- 外部变化通过事件通知，而不是让宿主强依赖内部 DOM

这个边界清晰，组件的可维护性会明显更高。

## 九、样式系统中的几个关键点

### 1. `:host`

` :host ` 用来选中组件宿主本身，也就是自定义标签这一层：

```css
:host {
  display: inline-block;
}
```

如果没有这层控制，自定义元素默认还是普通的未知标签表现，有时布局会比较难控。

### 2. `:host(...)`

可以根据宿主属性切换样式：

```css
:host([disabled]) {
  opacity: 0.5;
  pointer-events: none;
}
```

这对组件状态样式很有用。

### 3. `::slotted()`

用于给插槽传入的内容做有限样式控制：

```css
::slotted(h3) {
  margin: 0;
  color: #111827;
}
```

要注意它不是完整的深层选择器能力，只能命中被插入插槽的那一层节点，不能像普通 CSS 那样无限向下选。

### 4. `::part()`

如果你希望外部能定制组件内部某些区域样式，又不想完全打破封装，可以用 `part` 暴露样式接口：

组件内部：

```html
<button part="button">提交</button>
```

组件外部：

```css
my-button::part(button) {
  background: #2563eb;
  color: #fff;
}
```

这是一种比“直接开放内部类名”更稳的设计方式。

## 十、如何和 React、Vue 等框架配合

Web Components 不是要替代 React 或 Vue。现实情况通常是：它作为底层组件能力，被上层框架消费。

### 1. 在 React 中使用

React 可以直接渲染自定义标签：

```jsx
<user-card name="Zhang San" status="online" />
```

但有两个细节要注意：

- React 对自定义事件不是天然全兼容，很多时候需要手动 `addEventListener`
- 复杂对象传参更适合通过 `ref` 设置 property

例如：

```jsx
import { useEffect, useRef } from "react";

function Demo() {
  const ref = useRef(null);

  useEffect(() => {
    const el = ref.current;

    el.user = {
      name: "张三",
      role: "Frontend Engineer"
    };

    const onViewProfile = (event) => {
      console.log(event.detail);
    };

    el.addEventListener("view-profile", onViewProfile);

    return () => {
      el.removeEventListener("view-profile", onViewProfile);
    };
  }, []);

  return <user-card ref={ref} status="online" />;
}
```

### 2. 在 Vue 中使用

Vue 3 对自定义元素兼容性相对更自然，标签直接用即可。如果是复杂对象，同样推荐通过 `ref` 或封装适配层来处理。

如果一个团队大量使用 Web Components，通常会再包装一层 Vue / React 适配组件，把：

- 事件映射
- 属性同步
- 类型声明

统一收敛掉，这样业务层用起来会顺手很多。

## 十一、Web Components 适合哪些场景

不是所有组件都适合用 Web Components 来写，但下面这些场景，它往往有明显优势。

### 1. 跨技术栈通用组件库

比如公司内部多个团队分别使用不同框架，但又需要共享：

- 选择器
- 上传器
- 数据展示卡片
- 业务弹窗

这时 Web Components 很适合作为公共实现层。

### 2. 可嵌入第三方页面的控件

例如：

- 评论组件
- 在线客服入口
- 支付按钮
- 埋点可视化配置面板

这类组件通常要嵌入别人的页面，最怕宿主环境复杂、CSS 冲突严重。Shadow DOM 的收益会很直接。

### 3. 设计系统底层原子组件

如果设计系统希望未来不绑定某个单一框架，那么用 Web Components 做底层能力层，是一种很稳的路线。

### 4. 微前端共享基础组件

微前端场景里，各子应用的依赖、构建方式、框架版本可能都不一致。Web Components 在这里能起到很好的“公共协议层”作用。

## 十二、它不适合哪些场景

讲优点的同时，也要把边界说清楚。

### 1. 强依赖某个框架生态能力的复杂页面组件

比如一个组件大量依赖 React Hooks、Vue 响应式系统、上下文注入、路由能力，那么单独抽成 Web Components 反而可能增加复杂度。

### 2. 团队主要诉求是开发效率，而不是跨框架复用

如果整个团队技术栈很统一，而且短期内没有跨框架共享诉求，那么直接使用现有框架组件体系，通常会更省成本。

### 3. 对服务端渲染和同构链路要求特别高的场景

虽然现在 Web Components 也能配合 SSR 做一些工作，但整体心智负担和集成复杂度，通常还是高于纯框架内方案。

## 十三、工程落地时几个常见问题

### 1. 不要在 `constructor` 里做过重逻辑

构造函数只做初始化。真正依赖 DOM 环境、布局信息、外部属性的逻辑，尽量放在 `connectedCallback` 或更明确的渲染流程里。

### 2. 重新渲染时注意事件重复绑定

很多初学者写法是每次 `render()` 都直接重写 `innerHTML`，然后忘了处理旧节点上的事件，这会带来维护问题。要么做好解绑，要么采用更稳定的渲染策略。

### 3. 属性变化和内部状态更新要避免死循环

例如在 `attributeChangedCallback` 里又去改同一个 attribute，很容易触发重复更新。接口设计上要尽量单向、明确。

### 4. 注意可访问性

自定义组件不能只顾功能，还要补齐语义和可访问性，例如：

- 合理使用 `role`
- 管理焦点
- 支持键盘操作
- 提供必要的 `aria-*` 属性

如果你做的是输入类、弹层类、选择器类组件，这部分尤其不能省。

### 5. 表单组件要提前考虑交互协议

如果要做自定义输入框、选择框、开关等，最好尽早考虑：

- 如何和原生表单协同
- 值如何提交
- 校验如何暴露
- disabled、readonly、required 如何映射

这类组件后面往往会用到更深入的表单关联能力，不建议先随便写，后面再返工。

## 十四、和框架组件相比，应该怎么理解它的位置

我更倾向把 Web Components 看成“浏览器原生组件协议”，而不是“另一个要和 React、Vue 竞争的 UI 框架”。

它最有价值的地方不在于替代一切，而在于提供一个更底层、更稳定、更通用的封装方式。

可以这样理解：

- React / Vue 解决的是应用开发效率和状态管理问题
- Web Components 解决的是跨环境分发、封装边界、标准化组件接口问题

这两者并不冲突，很多成熟团队本来就是组合使用。

## 十五、总结

如果只把 Web Components 理解成“浏览器也能写组件了”，那还是太浅了。它真正重要的地方，在于它把“组件”从框架内部概念，推进成了浏览器可以直接理解和承载的标准能力。

它的核心价值主要体现在三点：

- 自定义元素，提供标准化组件入口
- Shadow DOM，提供真正可靠的封装边界
- Slot 与模板机制，提供灵活的内容组织方式

在实际项目里，它不一定是默认首选，但在跨框架组件库、设计系统、微前端共享能力、第三方嵌入式组件这些场景中，Web Components 的价值非常稳定，而且很难被完全替代。

如果你的目标只是快速搭页面，它未必是成本最低的方案；但如果你做的是基础设施层组件、长期维护的通用能力，Web Components 很值得认真掌握。
