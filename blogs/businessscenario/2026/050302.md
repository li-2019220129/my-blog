前端组件化说了很多年，但只要项目一大，大家最后总会遇到同一类问题：样式互相污染、DOM 结构被外部误操作、组件边界不够稳定。

很多时候我们以为自己已经把组件“封装”好了，实际上只是把代码写进了某个目录，或者借助框架做了语法层面的约束。真正到了复杂页面、跨团队复用、第三方页面嵌入这些场景，封装是否可靠，很快就会暴露出来。

`Shadow DOM` 的价值，正是在这里体现出来的。

它不是一个新框架，也不是某种工程约定，而是浏览器原生提供的一种 DOM 封装机制。它最大的意义在于：让一个组件内部的结构和样式，拥有相对独立、稳定、可控的边界。

## 一、Shadow DOM 是什么

简单来说，`Shadow DOM` 是挂载在普通 DOM 元素之上的一棵“隐藏子树”。这棵子树不直接暴露在页面主 DOM 结构中，而是作为宿主元素的内部结构存在。

例如：

```html
<user-card></user-card>
```

这个 `user-card` 可以作为宿主元素，然后在它内部再挂一棵 Shadow Tree：

```js
const host = document.querySelector("user-card");
const shadowRoot = host.attachShadow({ mode: "open" });

shadowRoot.innerHTML = `
  <style>
    .title {
      color: #1f2937;
    }
  </style>
  <h3 class="title">用户信息</h3>
`;
```

此时这个 `h3.title` 并不会直接出现在页面主文档那棵 DOM 树中，而是存在于 `user-card` 的 Shadow DOM 内部。

你可以把它理解成：浏览器在一个元素内部，再开辟了一层相对独立的 DOM 作用域。

## 二、为什么需要 Shadow DOM

如果只是做普通页面，很多人一开始会觉得：我用 `class` 命名规范、CSS Modules、`scoped` 样式，不也能解决大部分问题吗？

确实可以解决一部分，但 Shadow DOM 解决的是更底层的问题。

### 1. 样式隔离

先看一个最常见的问题。

页面里有两套模块都用了 `.title`：

```css
.title {
  color: red;
  font-size: 24px;
}
```

如果某个组件内部也写了：

```html
<h3 class="title">标题</h3>
```

那么只要样式范围没控制好，这个组件内部就很容易被宿主页面污染。尤其是你把组件投放到别的系统、管理后台、CMS 页面、低代码平台里时，外部 CSS 通常不会那么“讲道理”。

而 Shadow DOM 的样式隔离是浏览器级别的：

- 外部样式默认不会直接影响 Shadow Tree 内部
- Shadow Tree 内部样式默认也不会污染外部页面

这对组件稳定性帮助非常大。

### 2. 结构封装

有些组件内部 DOM 结构是实现细节，不应该被外部随意依赖。比如：

- 弹窗的内部包裹层
- 下拉面板的定位节点
- 复杂输入框的状态节点
- 业务卡片的装饰层

如果这些结构完全暴露在主 DOM 里，外部就很容易写出类似这样的代码：

```js
document.querySelector(".inner-wrapper").remove();
```

或者：

```css
.inner-wrapper {
  overflow: visible !important;
}
```

这类隐式依赖一旦形成，组件后面几乎很难重构。Shadow DOM 相当于给内部实现加了一层天然边界。

### 3. 更适合做可分发组件

如果一个组件要放进多个产品、多个技术栈甚至第三方站点里运行，那么“宿主环境不可控”是常态。Shadow DOM 在这种场景下的价值，比在单一应用内部更大。

## 三、Shadow DOM 的基本结构

Shadow DOM 里有几个核心概念，先分清楚会更容易理解后面的行为。

### 1. Host

宿主元素，也就是挂载 Shadow Root 的那个元素。

例如：

```html
<user-card></user-card>
```

这里的 `user-card` 就是 host。

### 2. Shadow Root

通过 `attachShadow()` 创建出来的根节点。

```js
const shadowRoot = host.attachShadow({ mode: "open" });
```

它是 Shadow Tree 的入口。

### 3. Shadow Tree

挂在 `shadowRoot` 下面的整棵 DOM 子树。

```js
shadowRoot.innerHTML = `
  <div class="card">
    <h3>标题</h3>
    <p>描述</p>
  </div>
`;
```

### 4. Light DOM

相对于 Shadow DOM，宿主元素原本所在的那一侧 DOM，经常被称为 `Light DOM`。

例如：

```html
<user-card>
  <span>外部传入的内容</span>
</user-card>
```

这里写在标签之间的内容，默认属于 Light DOM。

后面讲 `slot` 时，这个概念会非常重要。

## 四、如何创建一个 Shadow DOM

创建方式很直接：

```js
const host = document.querySelector("#box");
const shadowRoot = host.attachShadow({ mode: "open" });

shadowRoot.innerHTML = `
  <style>
    p {
      color: steelblue;
    }
  </style>
  <p>Hello Shadow DOM</p>
`;
```

HTML：

```html
<div id="box"></div>
```

这时 `#box` 内部会渲染出一段独立内容，但这段内容不是普通子节点意义上的“直接展开”，而是进入 Shadow Root 内部。

## 五、`mode: open` 和 `mode: closed` 的区别

创建 Shadow Root 时最常见的配置就是这个：

```js
host.attachShadow({ mode: "open" });
```

### 1. open

`open` 模式下，外部可以通过：

```js
host.shadowRoot
```

拿到这个 `shadowRoot`，因此调试、测试、局部操作都比较方便。

### 2. closed

`closed` 模式下：

```js
host.shadowRoot
```

会得到 `null`，外部无法直接访问这棵 Shadow Tree。

### 3. 实际项目里怎么选

大多数业务场景更常用 `open`，原因很现实：

- 更方便调试
- 更方便自动化测试
- 出问题时更容易排查

`closed` 更像是一种“进一步限制外部访问”的手段，但它也不是绝对安全机制。通常不建议为了“神秘封装”而盲目使用 `closed`，否则维护成本会升高。

## 六、样式隔离到底隔离到了什么程度

这是 Shadow DOM 最核心、也最容易被误解的部分。

### 1. 内部样式默认不会泄漏到外部

```js
shadowRoot.innerHTML = `
  <style>
    p {
      color: red;
    }
  </style>
  <p>内部段落</p>
`;
```

这里的 `p { color: red; }` 只会影响当前 Shadow Root 里的 `p`，不会把页面外部所有 `p` 都染红。

### 2. 外部样式默认也不会直接污染内部

如果主文档里有：

```css
p {
  font-size: 30px;
  color: green;
}
```

它通常不会直接影响 Shadow DOM 内部的 `p`。

### 3. 但不是所有东西都绝对隔离

这点要特别注意。Shadow DOM 并不是说“组件内部和外部彻底断绝关系”，有些内容依然会继承或共享，例如：

- 字体相关属性
- 某些 CSS 自定义属性
- 宿主元素本身的尺寸和布局上下文

比如外部页面设置了：

```css
body {
  font-family: "PingFang SC", sans-serif;
}
```

Shadow DOM 内部很多文本依然可能继承这个字体。如果你希望组件完全掌控视觉表现，通常要在内部显式声明关键样式，而不是完全依赖默认继承。

## 七、`:host`、`::slotted()`、`::part()` 这些选择器到底怎么理解

Shadow DOM 的样式隔离很强，但也意味着“内外如何有控制地交流”会变得更重要。这里最常用的几个选择器，基本就是为这个问题服务的。

### 1. `:host`

` :host ` 用来选中宿主元素本身，也就是自定义标签这一层。

```css
:host {
  display: block;
  border-radius: 12px;
}
```

如果你的组件是 `<user-card>`，那么 `:host` 实际上就是在给 `<user-card>` 本身加样式。

### 2. `:host(...)`

可以根据宿主状态或属性切换样式：

```css
:host([disabled]) {
  opacity: 0.5;
  pointer-events: none;
}
```

这对组件状态样式特别有用，因为很多状态最终都挂在宿主元素上。

### 3. `::slotted()`

当外部内容通过 `slot` 投递进 Shadow DOM 后，可以用 `::slotted()` 对这些被投递的节点做样式控制。

例如：

```html
<app-panel>
  <h3 slot="title">订单详情</h3>
</app-panel>
```

组件内部：

```html
<slot name="title"></slot>
```

样式：

```css
::slotted(h3) {
  margin: 0;
  color: #111827;
}
```

要注意，`::slotted()` 能控制的是“被插入插槽的那一层元素”，不能当作万能深层选择器来用。

### 4. `::part()`

如果组件作者想让外部安全地定制内部某些节点样式，可以通过 `part` 暴露部件。

组件内部：

```html
<button part="button">保存</button>
```

组件外部：

```css
save-button::part(button) {
  background: #2563eb;
  color: #fff;
}
```

这比直接暴露内部类名更稳，因为你暴露的是“样式接口”，而不是实现细节。

## 八、Shadow DOM 和 slot 的关系

如果没有 `slot`，Shadow DOM 的封装会非常强，但同时也会很“封死”。很多组件都需要一部分内容由外部传入，这就是 `slot` 存在的意义。

### 1. 默认插槽

组件内部：

```html
<div class="panel">
  <slot></slot>
</div>
```

外部：

```html
<app-panel>
  <p>这里是外部传入的内容</p>
</app-panel>
```

### 2. 具名插槽

组件内部：

```html
<header>
  <slot name="title"></slot>
</header>
<main>
  <slot></slot>
</main>
```

外部：

```html
<app-panel>
  <span slot="title">标题区域</span>
  <p>正文内容</p>
</app-panel>
```

### 3. 为什么它重要

因为 Shadow DOM 的目标不是把组件做成黑盒砖头，而是在保持边界稳定的前提下，仍然允许合理扩展。`slot` 就是这层扩展能力。

## 九、事件在 Shadow DOM 里有什么特别之处

很多人第一次接触 Shadow DOM，最容易忽略的不是样式，而是事件行为。

### 1. 事件重定向

当事件从 Shadow DOM 内部向外传播时，浏览器会做 `retargeting`，也就是事件目标在跨越边界时可能被“重定向”。

这意味着外部监听到的 `event.target`，不一定是内部真实点击的那个按钮节点，而可能是宿主元素本身。

这是浏览器故意为之，因为 Shadow DOM 的目标之一，就是隐藏内部实现细节。

### 2. 自定义事件要记得 `composed: true`

如果组件内部要向外发事件，通常这样写：

```js
this.dispatchEvent(
  new CustomEvent("confirm", {
    detail: { value: "ok" },
    bubbles: true,
    composed: true
  })
);
```

这里：

- `bubbles: true` 允许事件冒泡
- `composed: true` 允许事件穿过 Shadow DOM 边界

如果少了 `composed: true`，外层宿主代码可能根本接不到这个事件。

## 十、一个完整一点的 Shadow DOM 组件示例

下面写一个简单的面板组件，带标题插槽、内容插槽和关闭事件，比较接近真实项目里的写法。

```js
class AppPanel extends HTMLElement {
  constructor() {
    super();
    this.attachShadow({ mode: "open" });
    this.handleClose = this.handleClose.bind(this);
  }

  connectedCallback() {
    this.render();
    this.shadowRoot
      .querySelector(".close-btn")
      ?.addEventListener("click", this.handleClose);
  }

  disconnectedCallback() {
    this.shadowRoot
      .querySelector(".close-btn")
      ?.removeEventListener("click", this.handleClose);
  }

  handleClose() {
    this.dispatchEvent(
      new CustomEvent("close", {
        bubbles: true,
        composed: true
      })
    );
  }

  render() {
    this.shadowRoot.innerHTML = `
      <style>
        :host {
          display: block;
        }

        .panel {
          border: 1px solid #e5e7eb;
          border-radius: 12px;
          background: #fff;
          overflow: hidden;
        }

        .header {
          display: flex;
          justify-content: space-between;
          align-items: center;
          padding: 14px 16px;
          border-bottom: 1px solid #f1f5f9;
          font-weight: 600;
          color: #111827;
        }

        .body {
          padding: 16px;
          color: #475569;
          line-height: 1.7;
        }

        .close-btn {
          border: 0;
          background: transparent;
          cursor: pointer;
          font-size: 14px;
          color: #64748b;
        }

        ::slotted(h3) {
          margin: 0;
          font-size: 16px;
        }
      </style>

      <section class="panel">
        <header class="header">
          <slot name="title">默认标题</slot>
          <button class="close-btn" part="close-button">关闭</button>
        </header>
        <div class="body">
          <slot></slot>
        </div>
      </section>
    `;
  }
}

customElements.define("app-panel", AppPanel);
```

使用方式：

```html
<app-panel id="panel">
  <h3 slot="title">影子 DOM 示例</h3>
  <p>这里是正文内容。</p>
</app-panel>

<script>
  document.getElementById("panel").addEventListener("close", () => {
    console.log("面板被关闭了");
  });
</script>
```

这个例子里，Shadow DOM 负责三件事：

- 保护内部结构和样式
- 通过 slot 接收外部内容
- 通过自定义事件向外通信

这就是它在工程里最常见的组合方式。

## 十一、Shadow DOM 的优势到底体现在哪

### 1. 封装边界稳定

内部实现不容易被外部代码误伤，组件后续重构成本更低。

### 2. 样式污染明显减少

尤其适合放到宿主环境不受控的页面里。

### 3. 更适合做底层可复用组件

对于设计系统、通用组件库、嵌入式控件来说，Shadow DOM 的稳定性很有价值。

### 4. 和标准组件体系天然兼容

它和 `Custom Elements`、`slot`、模板能力可以自然组合，形成完整的浏览器原生组件模型。

## 十二、它也有边界，不是用了就万事大吉

### 1. 调试成本比普通 DOM 略高

虽然现代浏览器已经支持得不错，但你仍然需要适应 Shadow Root 的调试方式、事件传播行为、样式边界。

### 2. 主题定制要提前设计

封装太强有时反而会让外部不好改样式，所以做组件库时，最好提前设计好：

- 哪些样式由内部固定
- 哪些通过 CSS 变量开放
- 哪些通过 `part` 暴露

否则接入方很容易陷入“用是能用，但改不动”的状态。

### 3. 对表单和复杂交互组件要更谨慎

如果你做的是输入框、日期选择器、下拉框这类复杂组件，除了样式隔离，还要考虑：

- 焦点管理
- 键盘交互
- 表单值同步
- 可访问性

这些问题不会因为用了 Shadow DOM 就自动消失。

## 十三、哪些场景特别适合用 Shadow DOM

### 1. 需要嵌入第三方页面的组件

比如：

- 评论组件
- 在线客服入口
- 可视化挂件
- 营销活动卡片

这类组件最怕 CSS 冲突，Shadow DOM 几乎是天然优势。

### 2. 跨项目共享基础组件

如果一套组件要被多个业务系统复用，Shadow DOM 可以明显减少“这个系统里正常，换个系统就样式炸了”的情况。

### 3. 设计系统底层原子能力

当你希望组件长期稳定、对宿主页面低耦合时，Shadow DOM 很值得纳入方案。

## 十四、总结

Shadow DOM 的本质，不是“让 DOM 变复杂”，而是让组件边界变清晰。

它最关键的价值主要有三点：

- 提供浏览器原生的结构封装
- 提供更可靠的样式隔离
- 为可分发、可复用组件建立稳定边界

如果你只是写单页应用里的几个普通业务块，它未必是最优先的工具；但如果你要做真正长期维护、跨环境复用、对宿主环境有防御能力的组件，Shadow DOM 几乎是绕不过去的一环。

理解它，不只是多学一个 API，而是把“组件封装”这件事，往更底层、更扎实的方向推进一步。
