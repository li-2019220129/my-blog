从 0 写一个可用的 Prettier CLI 工具

lizhiyang前端工程化Node.jsCLIPrettier工程化

很多人会用 prettier --write，但如果让你自己做一个类似的小工具，往往就会停在“能跑”这一步。

这篇文章不做玩具版，我们来写一个真正能放进项目、也能写进简历里的 Prettier CLI。

最终希望做到这样的效果：

my-prettier src/**/*.ts --write
my-prettier src/**/*.ts --check
my-prettier src

它需要具备这些能力：

  • 支持 glob
  • 支持 --check
  • 支持 --write
  • 支持读取 Prettier 配置
  • 支持忽略 .prettierignore
  • 默认行为合理
  • 可以作为 npm CLI 发布

先明确一下“可用”和“玩具”的区别

如果只是演示概念，几十行代码就够了。

但如果你真的想把它当成一个工程化作品，至少要处理下面这些问题：

  • 传了路径，但没有匹配到文件怎么办
  • 同时传了 --check 和 --write 怎么处理
  • 遇到不支持的文件类型怎么办
  • 怎么读取项目里的 .prettierrc
  • 怎么尊重 .prettierignore
  • 检查模式下退出码怎么返回，方便接入 CI

这几个点，才是一个 CLI 工具从“会写”到“能用”的分水岭。

项目结构

my-prettier-cli/
├── bin/
│   └── index.js
├── src/
│   └── formatter.js
└── package.json

安装依赖

npm init -y
npm install prettier fast-glob

这里我们只用两个核心依赖：

  • prettier：负责格式化
  • fast-glob：负责文件匹配

CLI 入口

先处理命令行参数。

bin/index.js

#!/usr/bin/env node

const process = require("process");
const { formatFiles } = require("../src/formatter");

async function main() {
  const args = process.argv.slice(2);

  if (args.length === 0) {
    console.error("请输入文件路径，例如: my-prettier src/**/*.ts");
    process.exit(1);
  }

  const check = args.includes("--check");
  const write = args.includes("--write");
  const patterns = args.filter((arg) => !arg.startsWith("--"));

  if (check && write) {
    console.error("--check 和 --write 不能同时使用");
    process.exit(1);
  }

  const mode = check ? "check" : "write";

  await formatFiles(patterns, {
    check: mode === "check",
    write: mode === "write",
  });
}

main().catch((error) => {
  console.error(error);
  process.exit(1);
});

这里有两个我建议你一开始就养成的习惯：

  1. CLI 主流程用 async main() 包起来，异常统一兜底。
  2. 默认行为给 write，这样用户直接执行 my-prettier src 就能工作。

核心格式化逻辑

接着写真正干活的部分。

src/formatter.js

const fs = require("fs/promises");
const path = require("path");
const fg = require("fast-glob");
const prettier = require("prettier");

async function formatFiles(patterns, options = {}) {
  const files = await fg(patterns, {
    absolute: true,
    onlyFiles: true,
    ignore: ["**/node_modules/**", "**/.git/**"],
  });

  if (files.length === 0) {
    console.log("没有匹配到任何文件");
    process.exit(1);
  }

  let hasUnformatted = false;
  let hasError = false;

  for (const file of files) {
    try {
      const fileInfo = await prettier.getFileInfo(file, {
        ignorePath: ".prettierignore",
      });

      if (fileInfo.ignored || !fileInfo.inferredParser) {
        continue;
      }

      const source = await fs.readFile(file, "utf8");
      const config = (await prettier.resolveConfig(file)) || {};

      const formatted = await prettier.format(source, {
        ...config,
        filepath: file,
      });

      const relativePath = path.relative(process.cwd(), file);

      if (options.check) {
        if (formatted !== source) {
          hasUnformatted = true;
          console.log(`未格式化: ${relativePath}`);
        } else {
          console.log(`已规范: ${relativePath}`);
        }
        continue;
      }

      if (options.write && formatted !== source) {
        await fs.writeFile(file, formatted, "utf8");
        console.log(`已格式化: ${relativePath}`);
      }
    } catch (error) {
      hasError = true;
      console.log(`处理失败: ${file}`);
      console.error(error.message);
    }
  }

  if (hasError) {
    process.exit(1);
  }

  if (options.check && hasUnformatted) {
    process.exit(1);
  }
}

module.exports = {
  formatFiles,
};

这版代码为什么更像“工程项目”

如果你对比很多入门版实现，会发现这版多做了几件重要的事。

1. 用 prettier.getFileInfo() 过滤无效文件

这个 API 很实用，它能帮我们判断：

  • 文件是不是被忽略了
  • 文件类型 Prettier 支不支持

这样就不会把二进制文件、图片，或者不受支持的文件也拿去格式化。

2. 用 resolveConfig() 自动读取项目配置

项目里常见这些配置文件：

  • .prettierrc
  • .prettierrc.json
  • prettier.config.js

如果不读取配置，你这个 CLI 其实只是“调用了 Prettier”，并没有真正融入项目。

3. 用退出码支持 CI

--check 模式最重要的一点，不是打印日志，而是：

process.exit(1)

这样 GitHub Actions、GitLab CI、Jenkins 才能知道当前检查失败了。

4. 输出相对路径，而不是绝对路径

这是个小细节，但很像真实项目。

绝对路径太长，不利于阅读；相对路径更适合终端输出，也更适合 CI 日志查看。

package.json 怎么配

{
  "name": "my-prettier-cli",
  "version": "1.0.0",
  "bin": {
    "my-prettier": "./bin/index.js"
  }
}

有了这段配置之后，这个包就可以暴露出一个命令：

my-prettier

本地调试 CLI

在项目根目录执行：

npm link

然后就可以像正常命令一样测试：

my-prettier src/**/*.js --write
my-prettier src/**/*.js --check
my-prettier src

这里我推荐把默认行为设计成“写入”，因为大多数人执行这个工具，本来就是为了自动修复格式。

再补两个很关键的细节

1. --check 和 --write 不要同时生效

很多初学者写 CLI 时，喜欢“用户传什么我就都执行”。

但 --check 和 --write 的语义其实是冲突的：

  • --check 是只检查，不改文件
  • --write 是直接改文件

这种时候应该显式报错，而不是偷偷帮用户做决定。

2. 没匹配到文件时要明确失败

如果用户输入了错误路径，而你的 CLI 安静结束，排查起来会很痛苦。

所以我更建议这里直接返回失败，让调用者尽快发现问题。

还可以怎么继续升级

如果你想把这个项目继续做强，可以沿着下面几条线扩展。

彩色输出

安装 chalk，让错误、成功、提示更直观：

npm install chalk

并发处理

文件很多时，可以引入 p-map 做并发格式化。

不过这一层不要太早做，先把正确性做稳，再谈性能。

缓存机制

可以记录文件 hash，跳过没有变化的文件。

这会更接近 ESLint 的 --cache 体验。

发布 npm

如果你真的发到 npm，这个项目就不只是练习了，而是一个公开作品。

对前端同学来说，这类项目很适合体现几个能力：

  • Node.js 基础
  • 命令行工具开发
  • 工程化思维
  • 对工具链的理解

为什么我很推荐你做这种项目

很多时候，前端简历上写的都是：

  • 会 Vue
  • 会 React
  • 会 TypeScript

这些当然重要，但它们很容易同质化。

而“自己做一个能工作的 CLI 工具”会更能体现你的上限，因为它说明你已经开始从“使用工具的人”切换到“构建工具的人”。

这类项目很适合继续往下延伸：

  • 写 README
  • 写使用示例
  • 补异常场景
  • 发 npm
  • 接入自己项目
  • 写成系列博客

最后

如果只停在“能把代码格式化”，这个项目只是一个练习。

但如果你把参数设计、错误处理、配置读取、忽略规则、CI 接入都补完整，它就会变成一个很不错的工程化作品。

前端成长到一定阶段，差距往往不在于你会不会用工具，而在于你能不能把工具做出来。

这一点，特别值钱。
